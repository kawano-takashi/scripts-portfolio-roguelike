## 事前分析

レビュー前にコードベース分析を実施する:

1. **テストプロジェクトを特定する**: `.Tests` で終わるプロジェクトを見つける（よくある場所: `test/`、`tests/`、ソースプロジェクトの隣）
2. **テストクラスの棚卸し**: すべてのテストクラスとそれがカバーする本番クラスを一覧にする
3. **カバレッジギャップのマッピング**: 対応するテストのない本番クラスまたはパブリックメソッドを特定する
4. **問題点の特定**: 下記のチェックリストを適用して違反を見つける

## テストメソッド健全性チェックリスト

- [ ] テスト名が `[MethodName]_[Scenario]_[ExpectedBehavior]` 規約に従っている
- [ ] テストが明確なフェーズ分離（`// Arrange`、`// Act`、`// Assert`）で AAA パターンを使用している
- [ ] テストがちょうど1つの Act（テスト対象の単一メソッド呼び出し）を含んでいる
- [ ] テストにロジック（`if`、`for`、`while`、`switch`）がない -- 複数ケースには `[Theory]` を使用する
- [ ] テストがマジック文字列や数値の代わりに名前付き定数を使用している
- [ ] テストが実装詳細ではなく観測可能な振る舞い（戻り値、状態、例外）をアサートしている
- [ ] テストがリフレクション経由でプライベートメソッドを呼び出していない
- [ ] テストがテスト実行順序に依存していない
- [ ] テストが適切な属性を使用している: 単一ケースには `[Fact]`、パラメータ化には `[Theory]`
- [ ] Test Double 変数がその役割を反映した名前を持っている: `stubRepo`、`mockNotifier`
- [ ] Mock の `Verify()` が内部メソッド呼び出しの検証ではなく、真の副作用にのみ使用されている

## テストクラス健全性チェックリスト

- [ ] テストクラス名が `{ClassUnderTest}Tests` に一致している
- [ ] テストクラスがテスト対象クラスと同じ名前空間/フォルダ構成にある
- [ ] コンストラクタのセットアップがクラス内のほとんどのテストに必要なオブジェクトのみを作成している
- [ ] ヘルパーメソッド（`Create*`、`Build*`）が繰り返されるオブジェクト構築に使用されている
- [ ] テスト間で共有されるミュータブルな状態がない（`static` なミュータブルフィールドがない）
- [ ] `IClassFixture<T>` が高コストなセットアップにのみ使用されている（シンプルなオブジェクト作成には使わない）
- [ ] テストクラスが複数の本番クラスをテストしていない（該当する場合は分割する）
- [ ] テストがクリーンアップを必要とするリソースを作成する場合、`IDisposable` が実装されている

## テストプロジェクト健全性チェックリスト

- [ ] テストプロジェクト名が `{SourceProject}.Tests` に一致している
- [ ] テストプロジェクトが参照するのは: ソースプロジェクト、テストフレームワークパッケージ、モッキングパッケージのみ
- [ ] ユニットテスト用のテストプロジェクトがインフラパッケージ（EF Core、HTTPクライアント）を参照していない
- [ ] 必要なパッケージが存在する: `Microsoft.NET.Test.Sdk`、`xunit`、`xunit.runner.visualstudio`
- [ ] テストプロジェクトの `.csproj` で `IsPackable` が `false` になっている
- [ ] テストプロジェクトのフォルダ構成がソースプロジェクトのフォルダ構成をミラーしている
- [ ] コードカバレッジが設定されている: `coverlet.collector` パッケージが存在する
- [ ] テストメソッドが外部リソース（データベース、ファイルシステム、ネットワーク）に依存していない
